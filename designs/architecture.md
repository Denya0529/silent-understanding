# 静默理解系统（Silent Understanding System）

> 不是推荐引擎，不是行为分析仪表盘。
> 是一个越来越懂你的搭档，背后的机制。

## 设计理念

用户感知到的：它越来越懂我了。
用户感知不到的：一套观察→沉淀→内化→校验的后台循环。

**类比**：好的助理不会拿着表格问你偏好。他默默观察，慢慢调整，直到某天你发现——他总是刚刚好。

## 文件结构

```
memory/
├── impressions.jsonl       # 原始观察信号（只追加）
├── understanding.md        # 沉淀后的理解（定期更新）
└── YYYY-MM-DD.md           # 日常记录（已有，不变）

# 启动时读取优先级
SOUL.md          → 我是谁
USER.md          → 你告诉我的（显性）
understanding.md → 我观察到的（隐性）
MEMORY.md        → 事件与知识
```

### 三者的关系

| 文件 | 本质 | 来源 | 示例 |
|------|------|------|------|
| USER.md | 你的自我介绍 | 你告诉我的 | "我在日本工作，用中文沟通" |
| understanding.md | 我对你的理解 | 我观察推断的 | "给方案时先给结论，他不喜欢铺垫" |
| SOUL.md | 我的性格底座 | 初始设定+自我演化 | "有话直说，不当复读机" |

**USER.md 是名片，understanding.md 是相处后的默契。**

## 第一层：观察记录

### 什么时候记

**一轮话题对话结束时**（不是每条消息），回顾本轮交互，提取1-3条**有推断价值的信号**。

> 为什么不是每条消息？单条消息太碎，缺乏上下文，难以推断。一轮话题结束时有完整的决策链、反馈方式、沟通节奏，才能提炼出有意义的信号。
>
> "一轮话题"的判断：话题自然切换、用户明确换方向（如"先放一放，我们讨论XXX"）、或 session 即将结束（memory flush 触发时）。

### 记什么

不记行为本身，记**行为说明了什么**。

```jsonl
{"date":"2026-03-01","signal":"让子agent去做xxx，没有要求确认","infer":"执行类任务倾向委派，不需要中间确认","dim":"沟通风格","confidence":0.6}
{"date":"2026-03-01","signal":"视频语速慢+画面抖动，直接指出具体问题","infer":"反馈时给具体问题而非模糊感受，可以直接改不用追问","dim":"反馈方式","confidence":0.7}
{"date":"2026-03-01","signal":"说'这个任务可以让子agent来做'","infer":"清楚哪些事该花主session精力，哪些该委派","dim":"效率意识","confidence":0.7}
{"date":"2026-03-01","signal":"对Memory-xxxx的分析，追问背后理念而非使用方法","infer":"更关心Why和设计思想，不是How-to","dim":"思维偏好","confidence":0.6}
```

### 字段说明

| 字段 | 含义 |
|------|------|
| date | 观察日期 |
| signal | 原始信号（发生了什么） |
| infer | 推断（这说明了什么） |
| dim | 维度分类（见下方） |
| confidence | 初始置信度 0.5-0.8（单次观察不超过0.8） |

### 维度分类（dim）

```
沟通风格     — 喜欢什么样的回复方式
决策模式     — 怎么做决定、委派 vs 参与
反馈方式     — 怎么表达满意/不满意
节奏习惯     — 什么时候活跃、消息密度
审美偏好     — 视觉/内容/风格倾向
思维偏好     — 关注What/Why/How的哪个层面
工作方式     — 并行/串行、计划/即兴
情绪信号     — 什么让他兴奋/烦躁/无聊
```

### 不记什么

- 项目事实（那是 MEMORY.md 的事）
- 技术决策（那是 daily notes 的事）
- 隐私敏感信息
- 单次偶发行为（要有推断价值才记）

## 第二层：沉淀理解

### 频率策略

```
Phase 1 — 初识期（前2周）
  频率：每2-3天
  目标：快速建立基础认知画像
  方法：遍历所有impressions，按维度归纳

Phase 2 — 稳定期（1-3个月）
  频率：每周一次（heartbeat中执行）
  目标：微调和补充
  方法：只看上周新增impressions，对照现有理解更新

Phase 3 — 成熟期（3个月后）
  频率：有显著新信号时才更新
  目标：捕捉变化
  方法：标记"与现有理解矛盾"的信号，触发更新

例外：认知修正事件 → 任何阶段立即更新
```

### 沉淀过程

1. 读取 `impressions.jsonl` 中待处理的条目
2. 按维度聚合，寻找**重复出现的模式**
3. 单条印象不足以沉淀（至少2-3次相似信号才归纳）
4. 写入 `understanding.md`，附置信度标记
5. 在 impressions 中标记已处理（追加 `"processed":"2026-03-01"`）

### understanding.md 格式

```markdown
# Understanding
> 最后更新：2026-03-01
> 阶段：初识期

## 沟通风格
- [0.8] 给方案先说结论，不要铺垫过程。他问为什么时再展开
- [0.7] 能一次说完不要分多条。信息密度高 > 消息数量少
- [0.6] "好的"基本就是真的认可，不需要二次确认

## 决策模式
- [0.7] 大方向自己定，执行细节倾向信任委派
- [0.7] 不满意时会指出具体问题，不会模糊说"再改改"

## 思维偏好
- [0.6] 更关心Why和设计思想，不是How-to操作步骤
- [0.6] 喜欢对比分析（A方案 vs B方案），不喜欢单一推荐

## 审美偏好
- [0.8] 暗色系、科技感、克制。反感过度卡通化
- [0.7] 信息图/技术内容保持英文原文，更有质感

## 节奏习惯
- [0.5] 周末下午可能是密集互动时段（样本不足）

## 工作方式
- [0.7] 喜欢子Agent并行处理，主session专注核心决策
- [0.6] 偏好"先做出MVP再改"而不是"讨论完美方案再动手"
```

### 置信度规则

```
初始值：
  单次观察 → 0.5-0.6
  两次相似 → 0.7
  三次以上 → 0.8
  用户显式确认 → 0.9

变化：
  再次被验证 → +0.05（上限0.95）
  与行为矛盾 → -0.15
  30天无相关信号 → -0.05（下限0.3）
  降到0.3以下 → 标记为[存疑]，不删除，等待新信号

不会到1.0 — 人会变，永远留修正空间。
```

## 第三层：自然内化

### 加载时机

每次 session 启动时，AGENTS.md 已要求读取 SOUL.md + USER.md + MEMORY.md。

新增：**同时读取 understanding.md**（如果存在）。

### 如何影响回应

没有 if-else 规则。understanding.md 作为上下文的一部分被LLM自然吸收。

举例：
- 读到"给方案先说结论" → 回复时自然先给结论
- 读到"执行细节倾向委派" → 不会每步都来问确认
- 读到"更关心Why" → 解释设计时多说理由，少说步骤

**关键**：不需要显式规则引擎。LLM天然能把这些上下文内化为回应风格。这正是System Prompt工程的本质。

### 不做什么

- ❌ 不在对话中说"根据我的观察，你倾向于..."
- ❌ 不展示置信度分数
- ❌ 不主动告知"我更新了对你的理解"
- ❌ 不把 understanding.md 的内容直接引用给用户看

## 第四层：静默校验

### 触发条件

当我基于 understanding 做了某个判断，但用户反应不符合预期时：

| 我的判断 | 用户反应 | 校验动作 |
|---------|---------|---------|
| 省略了解释，直接给结论 | "为什么？" | 该条理解 -0.15 |
| 没确认直接执行 | "等一下，先让我看看" | 该条理解 -0.15 |
| 用简短回复 | 不回了/换话题 | 可能OK，不调整 |
| 用简短回复 | "能详细说说吗？" | 该条理解 -0.10 |

### 校验记录

```jsonl
{"date":"2026-03-02","ref":"沟通风格#1","action":"correction","reason":"省略解释后用户追问为什么","adjustment":-0.15}
```

### 矛盾处理

当新信号与现有理解矛盾时：
1. 先判断是否**场景差异**（工作场景 vs 闲聊场景可能不同）
2. 如果是同场景矛盾，降低置信度但不立即删除
3. 连续3次矛盾 → 重写该条理解
4. 人会变 — 3个月前的理解可能不再适用，这是正常的

## 与现有系统的关系

```
SOUL.md              — 我是谁（底座，很少变）
USER.md              — 你告诉我的（显性，偶尔更新）
understanding.md     — 我观察到的（隐性，持续演化）  ← 新增
MEMORY.md            — 事件与知识（长期记忆）
memory/YYYY-MM-DD.md — 当天发生了什么（日志）
impressions.jsonl    — 原始观察信号（只追加）          ← 新增
```

不改变现有任何文件的职责。只新增两个文件，一个记原始信号，一个存沉淀后的理解。

---

## 设计原则

1. **用户无感** — 整套机制对用户完全透明，不打扰
2. **宁缺毋滥** — 不确定的不写进understanding，等更多信号
3. **允许遗忘** — 低置信度的理解会自然淡出，这是Feature
4. **允许犯错** — 理解错了不可怕，有校验机制修正
5. **不过度推断** — 一次行为不代表模式，至少2-3次才归纳
6. **人会变** — 所有理解都有时效性，不存在永恒真理
